# Deep Review — Agent & Org Architecture (synthorg, divisions, fleet registry)

## Scope

- `config/org-divisions.json` (5 divisions, closed loop, ledger stages)
- `config/fleet.json` (8 site entries, vercel mapping, venture fields)
- `config/agents.json` (5 agent runtimes)
- `apps/synthorg/**` — `agent/agent.ts`, `agent/instructions.md`, `agent/lib/{model,synth}.ts`, `agent/tools/{delegate_task,check_budget,record_ledger}.ts`, `agent/channels/eve.ts`, `agent/connections/{modal,neon}.ts`, `agent/skills/{lifecycle,eval-gates,deploy-checklist,omni-market}.md`, `agent/subagents/{data_harvester,field_operator,qa_benchmark,training_orchestrator}/{agent.ts,instructions.md,tools/*}`
- `specs/0006-unified-cli-and-tracing.md`, `specs/0007-fleet-registry-and-sites.md`, `specs/0012-synthetic-org-divisions-and-handoffs.md`

## Findings

### Division model coherence

- 🟢 All five divisions have complete ownership contracts — each has `owner`, `primarySystems`, `owns`, `doesNotOwn`, `deliversTo`, `expectsFrom`, `delivers`, and `sla`. No orphan division, no division without a backing system. `config/org-divisions.json:5-278`.
- 🟢 Data → Research → BD → Execution → Orchestration handoff chain is bidirectionally consistent for the main spine: `data.delivers.research` matches `research.expectsFrom.data`; `research.delivers.bd` matches `bd.expectsFrom.research`; `bd.delivers.execution` matches `execution.expectsFrom.bd`; `execution.delivers.orchestration` matches `orchestration.expectsFrom.execution`. `config/org-divisions.json:39-50, 98-111, 158-171, 217-230, 261-273`.
- 🟡 **Bidirectional asymmetry: BD expects holdout corpora from Data, but Data does not declare delivery to BD.** `bd.expectsFrom.data` lists "Holdout eval corpora not used in training (disjoint slice)" (`config/org-divisions.json:154-156`), but `data.deliversTo` is only `["research","orchestration"]` and `data.delivers` has no `bd` entry (`config/org-divisions.json:25, 39-50`). The holdout handoff is declared one-sided.
- 🟡 **Bidirectional asymmetry: Orchestration declares `deliversTo` BD and Execution, but neither lists Orchestration in `expectsFrom`.** `orchestration.deliversTo: ["data","research","bd","execution"]` (`config/org-divisions.json:260`), yet `bd.expectsFrom = {research, execution, data}` and `execution.expectsFrom = {bd, research, data}` — orchestration is absent from both (`config/org-divisions.json:144-156, 205-215`). Orchestration's "prioritized work orders" are thus an unsolicited input to BD/Execution in the contract.
- 🟢 Division ownership is non-overlapping. `doesNotOwn` lists on each division cleanly fence off: Data has no recipes/checkpoints; Research has no deploy without BD charter; BD has no recipe invention; Execution has no experimental recipes; Orchestration delegates all division deliverables. `config/org-divisions.json:20-24, 78-82, 138-142, 199-203, 257-259`.

### Fleet registry consistency

- 🟢 Every `appPath` in `config/fleet.json` resolves to a real directory on disk: `apps/control`, `apps/synthorg`, `apps/sites/{hub,dumbmodel,benchmark-lab,research-rag,training-console,finance-lab}` all exist with real Next.js apps. `config/fleet.json:48,79,115,141,178,218,254,281`.
- 🟢 Every site's `orgDivision` maps to a valid division id in `org-divisions.json`: orchestration (control, hub, synthorg), bd (dumbmodel, benchmark-lab, finance-lab), research (research-rag, training-console). `config/fleet.json:36,62,102,129,164,201,241,268`.
- 🟢 Venture/monetization/dataConsent fields are consistent across the 5 public-facing sites that have them (hub, dumbmodel, benchmark-lab, research-rag, finance-lab): all use the same `venture.{valueProp, cta.{label,href}, monetization, dataConsent}` shape. `config/fleet.json:86-94, 148-156, 185-193, 225-233, 288-296`. Internal sites (control, synthorg, training-console) correctly omit venture fields.
- 🟢 Vercel `siteProjects` maps exactly the 5 sites with domains (hub, control, benchmark-lab, research-rag, dumbmodel). Domainless entries (synthorg, training-console, finance-lab) are correctly excluded from Vercel project mapping. `config/fleet.json:15-21`.
- 🟡 **Spec 0007 §"Sites (Phase A active)" table is stale — lists 6 sites but `fleet.json` has 8.** Missing `training-console` (added to registry but never added to spec table). `specs/0007-fleet-registry-and-sites.md:40-47` vs `config/fleet.json:29-298`. AGENTS.md site fleet table has the same omission.
- 🟡 **`finance-lab` is marked `status: "active"` but is Phase B with no domain and no Vercel project.** `config/fleet.json:285` says active; AGENTS.md describes it as "stub — Phase B". The app has real pages (`apps/sites/finance-lab/app/simulate/[platform]/page.tsx`, waitlist) so it is more than a stub, but with `domain: null` and no Vercel project, "active" overstates deployment state. Recommend `status: "phase-b"` or `status: "preview"`.
- 🟡 **`training-console` is marked `status: "active"` with `domain: null` and no Vercel project.** `config/fleet.json:258`. Unclear whether this is deployed anywhere or only runs locally on port 3006. If internal-only observability, status should reflect that (e.g. `internal`).
- 🟡 **Repo URL drift**: `platform.repo` is `https://github.com/jcdavis131/henington-homes` (`config/fleet.json:7`) — the legacy name. The actual codename is `bluehenre`. Two `legacyRepo` fields also point at `henington-homes`/`agent-lasso`/`arxiv_exam_app` which is correct as legacy tags, but the canonical `platform.repo` should point at the current repo.

### Agent architecture (synthorg)

- 🟢 **Deterministic backbone is present and load-bearing.** `check_budget` gates compute on `budget.remainingUsd` (`apps/synthorg/agent/tools/check_budget.ts:14-17`); `check_gates` returns a deterministic promote/reject from `gates.allPassed` (`apps/synthorg/agent/subagents/qa_benchmark/tools/check_gates.ts:10-12`); `deploy_model` refuses without `operatorApproved: true` (`apps/synthorg/agent/subagents/field_operator/tools/deploy_model.ts:14-17`); `monitor_drift` flags drift via a hard `mean < 0.3` threshold (`apps/synthorg/agent/subagents/field_operator/tools/monitor_drift.ts:12`). The LLM layer decides *when* to call these, but the gate decisions themselves are deterministic. This matches the deterministic-core-LLM-judgment pattern.
- 🟢 **All tools route through the unified access layer.** Every tool imports `synthFor()` from `agent/lib/synth.ts` → `@synthaembed/synth-core`. No direct fetch/db in `agent/**`. Satisfies Spec 0006 acceptance #1. `apps/synthorg/agent/lib/synth.ts:11-15`.
- 🟡 **Subagent models are hardcoded to `anthropic/claude-sonnet-4.6` and bypass the local-first strategy.** The Chief of Staff `agent.ts` carefully resolves `SYNTH_LOCAL_LLM_BASE_URL` / `SYNTH_AGENT_MODEL` to enable free/local/open-weights routing (`apps/synthorg/agent/agent.ts:18-27`), but all four subagents hardcode `model: "anthropic/claude-sonnet-4.6"` (`apps/synthorg/agent/subagents/data_harvester/agent.ts:4`, `field_operator/agent.ts:4`, `qa_benchmark/agent.ts:4`, `training_orchestrator/agent.ts:4`). The "workers can run a smaller/cheaper local model" comment in `agent.ts:16` is not implemented. Local-first mode would route the director through Ollama but workers through a paid Anthropic model — defeats the cost strategy.
- 🟢 **Subagent charters are distinct and scoped.** data_harvester = collect/chunk/pair; training_orchestrator = recipe/launch/poll; qa_benchmark = eval/gate/promote-reject; field_operator = deploy/serve/monitor. No charter overlap. `apps/synthorg/agent/subagents/*/instructions.md`.
- 🟡 **Missing guardrail #1 (per-team allowlist): no subagent declares an `allowedTools` allowlist.** Each subagent directory exposes a `tools/` folder, but nothing in `agent.ts` or `instructions.md` enforces that data_harvester cannot call `deploy_model`, or that qa_benchmark cannot call `launch_training`. Eve may infer scope from instructions, but there is no machine-enforced allowlist. `apps/synthorg/agent/subagents/*/agent.ts` all return `defineAgent({description, model})` with no tools array.
- 🟡 **Missing guardrail #2 (bounded loops): no `maxIterations`/`maxSteps`/`maxRetries` config on any subagent.** The Chief of Staff instructions mention "Autonomous termination" as prose (`apps/synthorg/agent/instructions.md:28-30`), but no subagent declares a loop bound in code. A runaway training_orchestrator polling `check_run` could burn budget until `check_budget` trips — and `check_budget` is only consulted by the Chief, not by the polling subagent.
- 🟡 **Missing guardrail #3 (full transcripts): trace propagation is acknowledged-but-unwired.** `agent/lib/synth.ts:7-9` admits "Until that hook is wired, a per-process trace is used" and Spec 0006 §"Eve agent" lists `agent/instrumentation.ts` as TBD with "Eve session id → SYNTH_TRACE_ID not wired". No `agent/instrumentation.ts` file exists. Subagent transcripts are not guaranteed to land in the trace store under a single objective trace id.
- 🟡 **`record_ledger` stage enum does not match the canonical ledger stages.** `apps/synthorg/agent/tools/record_ledger.ts:10` accepts `["collect","train","applied_test","deploy"]`, but `org-divisions.json` `closedLoop.ledgerStages` is `["collect","chunk","pairs","train","eval","deploy","index","pilot","charter","feedback","omni_sim"]` (`config/org-divisions.json:292`). Eve cannot record `chunk`, `pairs`, `eval`, `index`, `pilot`, or `charter` stages — the exact stages Spec 0012 §8 says divisions must log.
- 🟡 **Spec 0006 acceptance #5 unmet:** "Lint forbids direct fetch/db in `agent/**` outside synth-core — ⏳ not in CI". No ESLint rule or review script enforces this. The current clean state is by convention, not by gate. `specs/0006-unified-cli-and-tracing.md:81`.

### Closed loop (Spec 0012)

- 🟢 **Phase A hill-climb is reflected in worker runtime.** `services/worker/main.py` runs the train→eval→deploy→index chain and logs the corresponding ledger stages. `services/worker/main.py:1, 19, 100-164`.
- 🟢 **Phase A+ BD/charter gate is wired in both worker and core-api.** Worker calls `handoffs.submit_bd_candidate` after gates pass (`services/worker/main.py:100-120`), checks `handoffs.charter_allows_deploy` before deploy (`services/worker/main.py:122`), and skips deploy with "awaiting BD charter" note when no charter (`services/worker/main.py:157-164`). `deploy_model` raises on missing charter when `require_charter` is true (`services/core-api/app/services/models_svc.py:98-105`). BD queue has 2 real candidates (`content/fleet/bd/queue.json:5-35`). Charter files exist for 4 sites (`config/recipes/{hub,research-rag,benchmark-lab,dumbmodel}.json`).
- 🟡 **No charter file for `training-console` or `finance-lab`.** training-console is a research-division observability surface and may not need a model charter, but the registry marks it `active` under `research` division. finance-lab is Phase B and has no charter — acceptable for Phase B but should be explicit. `config/recipes/` directory listing.

### Skills

- 🟢 `agent/skills/lifecycle.md` is concrete and scoped: 4 stages, names the owning subagent for each, references the ledger and budget gate. `apps/synthorg/agent/skills/lifecycle.md:1-13`.
- 🟢 `agent/skills/eval-gates.md` is concrete and scoped: 4 explicit gates with thresholds (effective rank > baseline, nDCG@10 ≥ baseline, Matryoshka tolerance, disjoint slices) and a "reject if unverifiable" rule. `apps/synthorg/agent/skills/eval-gates.md:1-14`.
- 🟢 `agent/skills/omni-market.md` is concrete and scoped: simulation-only guardrail, lists exact workerbee commands, division routing table, artifact paths, and the "Sharpe regresses" runbook. `apps/synthorg/agent/skills/omni-market.md:1-42`.
- 🟡 **`agent/skills/deploy-checklist.md` is a generic template, not a BHRE-scoped skill.** Uses `[Service/Release]`, `[Date]`, `[Name]` placeholders; references "If ~~source control~~ is connected" with strikethrough text and `~~CI/CD~~` — clearly copied from a generic skill library. Has a slash-command frontmatter (`/deploy-checklist $ARGUMENTS`) that does not match the other three skills' format (no frontmatter, prose guidance for Eve to load). `apps/synthorg/agent/skills/deploy-checklist.md:1-77`. Should be rewritten to reference `services/worker`, `core-api` deploy endpoints, the charter gate, and the actual rollback triggers used by BHRE.

## Risk

- **Subagent allowlists not enforced (🟡, severity: medium).** A misrouted LLM call could have `qa_benchmark` invoke `launch_training` or `data_harvester` invoke `deploy_model`. The deterministic gates (operator approval, charter) would still block irreversible harm, but budget could be spent by the wrong subagent with no division-aligned ledger stage. Compounds with the missing loop bound.
- **Subagent model override bypasses cost strategy (🟡, severity: medium-high).** Local-first mode (the documented zero-marginal-cost path) only applies to the Chief of Staff; the four workers always call Claude. A full hill-climb in local mode would still incur Anthropic spend for every worker turn, defeating the `RESEARCH_COST_CEILING_USD_PER_DAY` intent.
- **`record_ledger` stage enum drift (🟡, severity: medium).** Eve cannot write `chunk`, `pairs`, `eval`, `index`, `pilot`, or `charter` stages — so Orchestration's stall detector (Spec 0012 §8: "`pairs` without `train` > 48h → escalate") cannot be fed by Eve. The Python worker can write these, but the agent layer cannot, creating a gap in cross-division signals when Eve drives the loop.
- **Trace id not propagated from Eve sessions (🟡, severity: medium).** Without `agent/instrumentation.ts`, subagent transcripts land under per-process traces instead of one objective trace id. Spec 0006 acceptance #2 and #4 stay partial. Auditability of a full hill-climb is weakened.
- **BD/Execution do not expect Orchestration input (🟡, severity: low).** Orchestration's "prioritized work orders" are not in BD/Execution's `expectsFrom`, so a contract reader could conclude those work orders are unsolicited. In practice Eve routes them anyway, but the registry disagrees with the runtime.
- **`finance-lab` and `training-console` status "active" with no domain/Vercel (🟡, severity: low).** Operators reading `pnpm fleet list` may assume production deployment. finance-lab is Phase B simulation-only; training-console is observability. Status should distinguish "deployed" from "internal/preview".
- **Spec 0007 site table stale (🟡, severity: low).** Documentation drift — agents reading the spec instead of the registry will miss training-console. Compounds registry-as-source-of-trust goal.

## Recommendation

Ordered by priority:

1. **Fix subagent model resolution (high).** Replace each `apps/synthorg/agent/subagents/*/agent.ts` hardcoded `model: "anthropic/claude-sonnet-4.6"` with a shared resolver (e.g. import `resolveFromEnv` from `agent/lib/model.ts` keyed by subagent name, so `SYNTH_MODEL_DATA_HARVESTER` etc. can override per worker). Default to the Chief of Staff's resolved model when unset.
2. **Add per-subagent tool allowlists (high).** In each `agent.ts`, pass `tools: [...]` explicitly importing only that subagent's tools. data_harvester → `ingest_corpus, lmar_chunk, synth_pairs`; training_orchestrator → `launch_training, check_run`; qa_benchmark → `run_eval, check_gates`; field_operator → `deploy_model, monitor_drift`. This makes the division scope machine-enforced.
3. **Align `record_ledger` stage enum with `org-divisions.json` `closedLoop.ledgerStages` (high).** Expand the zod enum to `["collect","chunk","pairs","train","eval","deploy","index","pilot","charter","feedback","omni_sim"]` (or import from a shared schema). Lets Eve emit the same stages as the Python worker so Orchestration's stall detector works on agent-driven runs too.
4. **Wire Eve session id → `SYNTH_TRACE_ID` (medium).** Add `apps/synthorg/agent/instrumentation.ts` per Spec 0006's TBD, set `SYNTH_TRACE_ID` from the eve session id at handoff, and verify child spans inherit. Closes Spec 0006 acceptance #2 and #4.
5. **Add bounded-loop config to subagents (medium).** Pass `maxSteps`/`maxRetries` to `defineAgent` for each subagent (e.g. training_orchestrator polling cap, qa_benchmark eval retry cap). Enforces the "Autonomous termination" prose in `instructions.md`.
6. **Resolve bidirectional handoff asymmetries in `org-divisions.json` (medium).** Either (a) add `bd` to `data.deliversTo` and a `data` entry under `data.delivers` for holdout corpora, or (b) remove `data` from `bd.expectsFrom` if the holdout is actually owned by BD. Add `orchestration` to `bd.expectsFrom` and `execution.expectsFrom` to match `orchestration.deliversTo`.
7. **Rewrite `agent/skills/deploy-checklist.md` to be BHRE-scoped (medium).** Replace generic placeholders with the real deploy path: worker `deploy_model`, `SYNTH_CHARTER_GATE`, `POST /v1/admin/deploy`, charter file requirement, eval-gate agreement, and rollback triggers tied to nDCG drop / p95 latency on `core-api`.
8. **Correct site status semantics in `fleet.json` (low).** Change `finance-lab` to `status: "phase-b"` (or `preview`) and `training-console` to `status: "internal"` (or document where it deploys). Update Spec 0007 §"Sites (Phase A active)" table and AGENTS.md site fleet table to include `training-console`.
9. **Fix `platform.repo` URL in `fleet.json` (low).** Point at the current `bluehenre` repo, keep `henington-homes` only in `legacyRepo` fields where it belongs.
10. **Add ESLint rule forbidding `fetch`/`pg`/direct DB imports in `apps/synthorg/agent/**` outside `lib/synth.ts` (low).** Closes Spec 0006 acceptance #5; makes the current clean-by-convention state enforceable in CI.
