# Spec 0021 — Monetization layer: diversified passive + active income, low overhead

- **Status:** Draft (Operator-directed 2026-07-03)
- **Owner:** Cursor (commerce wiring, site CTAs) · Claude (billing backend, core-api metering)
- **Refines:** Spec 0015 (venture fleet) · Spec 0019 (corporate topology) · Spec 0018 (refinery)
- **Related specs:** 0004 (core-api) · 0008 (eval-harness) · 0013 (simulation guardrail)

## Problem

The platform's autonomous operating loop (collect → train → validate → serve)
already produces sellable assets automatically: trained ASN models, curated OKF
datasets, eval scorecards, and live API inference. But the current revenue model
(Spec 0015/0019) is weighted toward **active** streams — services and
engagements that require per-customer human delivery (sprints, managed tenants,
enterprise deployments, custom harvests). These are high-margin but high-overhead
and don't scale without headcount.

The vision is missing the **passive** half: streams where the loop produces the
asset, the platform lists it, the customer self-serves, and billing is automated.
No human touches a transaction. Scale-to-zero infrastructure (Modal GPU,
Vercel/edge, Railway) means idle overhead is zero — revenue flows while the
loop runs unattended.

## Goals

- Define a **diversified income portfolio** with both passive (automated,
  zero-per-transaction-overhead) and active (human-delivered, high-margin)
  streams, each mapped to a business unit.
- Establish the **low-overhead principle**: passive streams require no human
  per-transaction; the loop produces, the platform lists, the customer
  self-serves, billing is automated (Stripe metered or Medusa one-time).
- Wire the missing passive stream infrastructure: metered API billing,
  automated certification, dataset access tiers, hosted inference tier.
- Make active streams **feed passive streams** — every engagement should
  terminate in a recurring/automated revenue path (sprint → managed → metered
  API; certification → badge renewal; harvest → premium dataset subscription).

## Non-goals

- Live trading revenue (Spec 0013 wall — simulation only in v1).
- Advertising / sponsored content (violates enterprise B2B voice, Spec 0017/voice doc).
- Multi-currency / multi-region tax compliance beyond what Medusa + Stripe
  already handle (out of scope until a non-USD customer closes).
- Replacing Medusa with a different commerce backend (it's live and working).

## Design

### The portfolio

Two tiers, six passive + five active streams, each owned by a business unit.

#### Passive streams (automated, zero per-transaction overhead)

| # | Stream | BU | Asset the loop produces | Billing | Overhead |
|---|---|---|---|---|---|
| P1 | **Metered API inference** | company (storefront) | Trained ASN model served via core-api | Stripe metered — per-query or flat monthly w/ quota | Zero (scale-to-zero GPU; billing is a webhook) |
| P2 | **Hosted domain-model tier** | company (storefront) | Domain-tuned model hosted at edge (Modal serverless GPU) | Monthly subscription (Medusa recurring) | Zero (model already trained by loop; Modal scales to zero) |
| P3 | **Dataset marketplace** | refinery (data) | OKF datasets from continuous builder | One-time per corpus (Medusa) + free preview tier | Zero (signed-URL delivery; builder runs unattended) |
| P4 | **Automated certification** | validation (bd) | Eval-harness scorecard + badge | Flat fee per submission (Medusa) | Zero (customer self-submits endpoint → harness runs → scorecard publishes → charge) |
| P5 | **Premium research notes** | signals (execution) | Auto-generated strategy reports from sim loop | Monthly subscription (Stripe) | Zero (reports generated by omni_loop; gated by paywall) |
| P6 | **Open-core model downloads** | company (storefront) | ASN models published to HuggingFace / GitHub | Free (goodwill/SEO/flywheel) → drives P1/P2 hosted revenue | Zero (publication is a CI step) |

#### Active streams (human-delivered, high-margin, feed passive)

| # | Stream | BU | Delivery | Terminates in |
|---|---|---|---|---|
| A1 | **Evaluation Sprint** | company | 2-week benchmark, fixed fee | → P1 metered API (customer wants ongoing inference) |
| A2 | **Managed Embeddings** | company | Dedicated tenant + monthly retraining | → P1/P2 (recurring, transitions to passive) |
| A3 | **Enterprise Platform** | company | Private deployment, annual | → P2 (hosted tier add-on) |
| A4 | **Custom harvest / prep** | refinery | Data engagement, lead → scoped work | → P3 (premium dataset subscription) |
| A5 | **Research engagement** | arxiviq | Custom corpus retrieval, lead → scoped work | → P1 (metered API on their corpus) |

### The low-overhead principle (normative)

1. **Every passive stream must be fulfillable without a human in the loop.**
   If a transaction requires a human to deliver, it's active, not passive —
   reclassify or automate the bottleneck.
2. **Billing is event-driven, not invoice-driven.** Stripe metered usage
   webhooks or Medusa checkout — no manual invoicing for passive streams.
3. **Idle cost is zero.** Passive infrastructure (Modal GPU, Vercel, Railway)
   must scale to zero when no customer is calling. Budget ceilings (already in
   core-api `/v1/budget`) enforce this per tenant.
4. **Active streams are the funnel, not the floor.** Every active engagement
   carries a termination clause: "this ends in a recurring/automated revenue
   path." If it doesn't, it's a one-off and the venture fleet isn't doing its job.
5. **Cross-BU referral is automatic.** Free tools (dumbmodel) link to paid
   next-steps (slasso certification, bhenre managed) with `source=<siteId>`
   attribution (already shipped for leads; extend to checkout metadata, Spec 0019 TOPO-002).

### Architecture

```
Autonomous loop (collect → train → validate → serve)
        │
        ├─ produces ──→ ASN model ──→ P1 metered API (core-api + Stripe)
        │                          ──→ P2 hosted tier (Modal + Medusa recurring)
        │                          ──→ P6 open-core publish (CI step)
        │
        ├─ produces ──→ OKF dataset ──→ P3 marketplace (Refinery + signed URLs)
        │
        ├─ produces ──→ Eval scorecard ──→ P4 automated cert (slasso self-service)
        │
        └─ produces ──→ Strategy report ──→ P5 premium notes (Signals paywall)

Active engagements (A1-A5) ──funnel──→ passive streams (P1-P6)
```

## Contract

### Metered usage event (P1)

```json
POST /v1/billing/usage (internal, called by core-api after each /v1/search or /v1/embed)
{
  "workspace_id": "ws_abc",
  "event": "embed_query",
  "quantity": 1,
  "model_version": "asn-v3.2",
  "timestamp": "2026-07-03T16:00:00Z"
}
```

core-api records the event; a Stripe metered subscription on the workspace's
customer ID accumulates and invoices on the billing cycle. Spend caps
(already in `/v1/budget`) hard-stop inference when exceeded.

### Certification submission (P4)

```json
POST /v1/certify/submit (public, after Medusa checkout)
{
  "order_id": "medusa_order_xyz",
  "model_endpoint": "https://customer.com/embed",
  "exam_set": "rag-multi-hop",
  "workspace_email": "team@customer.com"
}
```

Returns `{ "certification_id": "cert_abc", "status": "queued" }`. The worker
runs the eval-harness against the endpoint, publishes a scorecard at
`/scorecards/cert_abc`, and emails the badge + link.

## Data model

- `billing_usage_events` (new table): `id`, `workspace_id`, `event`,
  `quantity`, `model_version`, `created_at`. Indexed on `workspace_id + created_at`.
- `certifications` (new table): `id`, `order_id`, `model_endpoint`, `exam_set`,
  `status` (queued/running/passed/failed), `scorecard_slug`, `created_at`,
  `completed_at`.
- `stripe_customer_id` added to `workspaces` (migration).
- Medusa: new product collections — "API credits", "Hosted models",
  "Datasets", "Certification", "Premium notes".

## Acceptance criteria

1. A customer can generate an API key on the storefront, call `/v1/search`, and
   see usage accumulate on a Stripe metered invoice with zero human intervention.
2. A customer can submit a model endpoint for certification on slasso, pay via
   Medusa checkout, and receive a published scorecard + badge automatically.
3. A customer can subscribe to a hosted domain model on the storefront and
   receive an API key that routes to the Modal-hosted inference endpoint.
4. Refinery datasets have free-preview and paid-full tiers; paid checkout
   delivers a signed URL automatically.
5. Signals strategy reports behind a Stripe subscription paywall; free preview
   shows the first report, paid unlocks the feed.
6. Trained ASN models publish to HuggingFace via a CI step after each deploy gate
   pass (open-core).
7. The hq revenue dashboard shows per-stream MRR with a passive/active split and
   per-BU attribution.
8. Every active engagement CTA (pricing page, BU homepages) links to a passive
   termination path (sprint → metered API; certification → badge renewal).
9. Idle infrastructure cost is zero: Modal GPU scales to zero when no inference
   request is active; budget ceilings hard-stop inference per workspace.

## Rollout & rollback

- **Phase 1 (P1 + P4):** metered API billing + automated certification. These
  are the highest-leverage passive streams (core-api already serves; eval-harness
  already runs). Behind a `BILLING_ENABLED` flag per workspace.
- **Phase 2 (P3 + P5):** dataset marketplace + premium notes. Requires Refinery
  (Spec 0018) to ship first.
- **Phase 3 (P2 + P6):** hosted inference tier + open-core publishing. Requires
  Modal GPU (Spec 0011) to be provisioned.
- **Rollback:** `BILLING_ENABLED=false` reverts to free-tier inference (current
  state). Certification submissions queue but don't charge. No data loss —
  usage events remain recorded for back-billing.

## Risks

- **Stripe metering complexity** — metered billing has edge cases (overages,
  mid-cycle upgrades). Mitigation: start with flat monthly + hard quota (simpler),
  graduate to true metered after first 10 paying customers.
- **Certification fraud** — customer could submit a model that's actually a
  proxy to a commercial API. Mitigation: eval-harness runs against the endpoint
  with held-out queries; scorecard records latency/consistency fingerprints.
- **Dataset licensing** — OKF provenance must be airtight for paid datasets.
  Mitigation: Spec 0018 consent/licensing copy (Operator G3 gate) before any
  paid dataset ships.
- **Open-core cannibalization** — publishing models openly could reduce hosted
  inference revenue. Mitigation: open models are the funnel (SEO, trust,
  data flywheel); hosted tier sells convenience + SLA + the latest model, not
  the weights.
- **Scope creep into live trading** — P5 (premium notes) must stay simulation-only
  (Spec 0013). No trading advice, no real-capital signals. Mitigation: paywall
  copy explicitly says "paper-trading simulation reports, not investment advice."
