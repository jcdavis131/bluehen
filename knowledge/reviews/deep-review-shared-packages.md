# Deep Review — Shared Packages (fleet, ui-fleet, synth-core, eval-harness, runboard, datalab, omni-sim, asn-engine, cli)

## Scope
- **TS packages read in full**: `packages/fleet/src/{index,workspace-env,narrative,types}.ts` + `package.json`; `packages/ui-fleet/src/{index,FleetShell,PageHeader,SiteSubnav,FleetNavMobile,LiveSearchPanel,site-api,urls}.ts(x)` + `package.json` + `routes/index.ts`; `packages/synth-core/src/{client,index,trace}.ts` + `package.json`; `packages/cli/src/{index,parse-args}.ts` + `package.json`.
- **Python packages read in full**: `eval_harness/{__init__,gates,metrics,runner}.py` + `pyproject.toml`; `runboard/{__init__,run,store,telemetry,api}.py` + `pyproject.toml`; `datalab/{__init__,pipeline,store}.py` + `pyproject.toml`; `omni_sim/{__init__,sim_env,platforms,kag,smart_search}.py` + `pyproject.toml`; `asn_engine/__init__.py` + `pyproject.toml`.
- **Cross-cuts**: `pyproject.toml` (root workspace members); `services/core-api/pyproject.toml` + `app/services/eval.py`; repo-wide grep for direct Postgres / `synth-core` importers / `siteHref` / `FleetNavMobile` / `parse-args`.
- **Smoke-imports run**: `omni_sim`, `runboard`, `datalab`, `asn_engine`, `eval_harness` via `uv run --no-sync python -c "import <pkg>"`.

## Findings

- 🔴 **`eval-harness` is not a uv workspace member** — `pyproject.toml:9-17` lists `agentkit, asn-engine, datalab, omni-sim, runboard, core-api, trainer` but omits `packages/eval-harness`. Smoke-import `uv run --no-sync python -c "import eval_harness"` fails with `ModuleNotFoundError: No module named 'eval_harness'` while every other package imports clean. `eval-harness/pyproject.toml:13` itself declares `asn-engine = { workspace = true }`, so its own manifest is workspace-aware — the root just never enrolled it. Consumers work around it with `sys.path.insert(0, str(ROOT / "packages" / "eval-harness"))` (e.g. `scripts/tenant_baseline.py:24`, `scripts/engine_proof.py:27`) and the worker Dockerfile copies the pyproject by hand (`infra/Dockerfile.worker:8`). The deploy-gate package (spec 0008) is the load-bearing piece of the whole ML release flow and it isn't installable through `uv sync`.

- 🔴 **`services/core-api` imports `eval_harness` but doesn't declare it** — `services/core-api/app/services/eval.py:47` does `from eval_harness.runner import evaluate_checkpoint` inside `run_eval_for_workspace`. `services/core-api/pyproject.toml:6-13` lists `omni-sim, runboard, datalab` as workspace deps but **not** `eval-harness`. Combined with the workspace-membership gap above, the eval stage of core-api compiles only because of a path/Dockerfile coincidence, not a declared dependency. Any clean `uv sync` + `uv run uvicorn app.main:app` on a fresh checkout will throw `ModuleNotFoundError` the first time someone hits `/v1/eval/run`.

- 🟡 **`synth-core` is bypassed by every site** — the hard rule (`AGENTS.md`: "Sites → core-api only via `@synthaembed/synth-core`") is not enforced in code. `packages/ui-fleet/src/site-api.ts:33-51` implements `apiFetch` with a raw `fetch(${baseUrl}${path})` + `Authorization: Bearer ${apiKey}` header and **no trace spans**. Repo-wide grep shows zero site files import `@synthaembed/synth-core`; the only importer is `apps/synthorg/agent/lib/synth.ts`. `packages/cli/src/index.ts:32` is the other correct consumer. So `Synth`'s `withSpan`/`setTraceSink` instrumentation (the "every call is traced" promise in `client.ts:1-8`) is absent from all site→core-api traffic. The calls do reach core-api (not Postgres), so it's a chokepoint-skipping issue rather than a tenancy-bypass issue — but the trace store is blind to every site request.

- 🟡 **`mrlWithinTolerance` deploy gate is a hardcoded stub** — `packages/eval-harness/eval_harness/gates.py:13` returns `"mrlWithinTolerance": True` unconditionally. The other two gates are real (`rankAboveBaseline` from `effective_rank > BASELINE_EFFECTIVE_RANK=8.0`, `ndcgNonRegression` from `ndcg10 >= MIN_NDCG10=0.35`). Spec 0008 frames three gates; one of them never fails. `runner.py:66-67` then computes `allPassed = all(v is True for v in gates.values())`, so the MRL gate is a free pass on every deploy decision. No Matryoshka tolerance check is wired even though `Synth.model.deploy` accepts `truncateDims` (`synth-core/src/client.ts:85`).

- 🟡 **`eval-harness` declares `testpaths = ["tests"]` but has no tests directory** — `packages/eval-harness/pyproject.toml:26` vs. `Glob packages/eval-harness/tests/*.py` returns nothing. The other Python packages all have tests (`datalab/tests/test_datalab.py`, `test_watch.py`; `asn-engine/tests/test_spectral.py`, `test_sleep.py`; `omni-sim/tests/test_sim_env.py`, `test_smart_search.py`; `runboard/tests/test_runboard.py`; `agentkit/tests/test_agentkit.py`). The deploy-gate package is the one most in need of tests and is the only one with none.

- 🟡 **Dead code: `FleetNavMobile` and `parse-args.ts`** — `packages/ui-fleet/src/FleetNavMobile.tsx` (43 lines, well-built, accessible `aria-expanded` toggle) is **not** re-exported from `packages/ui-fleet/src/index.ts` (barrel has no `FleetNavMobile` entry) and grep finds zero importers across `apps/`. `packages/cli/src/parse-args.ts` exports `parseGlobalFlags` but `packages/cli/src/index.ts` never imports it and there are no other importers. Both are orphaned; the `use-design-system` refinement ("keep package's barrel export current") is violated by `FleetNavMobile`'s absence from the barrel even if it were alive.

- 🟡 **`StatusPill` lives in a single site, not in `ui-fleet`** — the prompt listed `StatusPill` under ui-fleet, but `Grep` for `StatusPill` resolves only to `apps/sites/training-console/components/StatusPill.tsx` (+ `RunDetail.tsx`). A status-pill is exactly the kind of primitive the design-system package should own (the prompt's expectation confirms the intuition). It's currently duplicated-risk territory: any other site that wants a status pill will reinvent it. Tokens (`tokens.css`) and `ApiStatusBanner` (`SiteSubnav.tsx:25-45`) live in ui-fleet, but the pill shape doesn't.

- 🟢 **`fleet` barrel is clean and accurate** — `packages/fleet/src/index.ts` re-exports every public symbol from `narrative.ts`, `bd-queue.ts`, `org-divisions.ts`, `types.ts` and the four site helpers (`getFleet, getPlatform, listSites, getSite, getSiteByDomain, devCommand, pairProgramContext`). `GLOSSARY`, `RE`, `BRAND`, `SITE_CIRCUIT`, `SITE_NAV`, `getSiteCircuit`, `getSiteNav`, `stageLabel`, `DIVISION_RELAY` are all present and consumed across `apps/sites/*` (verified by grep). `workspace-env.ts` correctly resolves `data/workspaces/{siteId}.env` → `SYNTH_API_BASE_URL` / `SYNTH_API_KEY` with the documented fallback chain (`workspace-env.ts:48-53`).

- 🟢 **`synth-core` is a single, well-shaped chokepoint** — `packages/synth-core/src/client.ts` exposes `Synth` with seven namespaces mirroring the lifecycle (`data, train, evals, model, vector, omni, ledger, trace`). Constructor wires `setTraceSink` back through the same client (`client.ts:32-34`), so trace spans ride the same uniform endpoint. `synthFromEnv` (`index.ts:19-23`) is the one-liner env constructor. `package.json` deps are minimal (`zod` + TS). The package itself is correct — the problem is the sites don't use it (see 🟡 above).

- 🟢 **`ui-fleet` route helpers are correctly shared** — `packages/ui-fleet/src/routes/index.ts` exports `GET_health, POST_search, POST_diagnose, GET_models, GET_ledger, POST_feedback, POST_hillClimb, GET_bdQueue`, and sites import them with the `as GET` / `as POST` rename pattern (verified: `apps/sites/benchmark-lab/app/api/search/route.ts:1`, `apps/sites/hub/app/api/ledger/route.ts:1`, `apps/sites/dumbmodel/app/api/diagnose/route.ts:1`, etc.). `package.json` `exports` map exposes `./routes` separately so client components don't pull server-only code into bundles. Good barrel hygiene for the route surface.

- 🟢 **`runboard` is well-isolated with opt-in extras** — `runboard/pyproject.toml:6-13` keeps core deps to `numpy, pydantic`; `fastapi, uvicorn` are behind the `api` extra; `pytest, httpx` behind `dev`. `store.py` defaults to `$RUNBOARD_DIR` then `repo_root()/data/runs` (`store.py:71`) and uses atomic `os.replace` for manifests (`store.py:83-85`). `telemetry.py` reimplements `effective_rank` in numpy with a docstring stating it matches `asn_engine.spectral.effective_rank` — and the formula (entropy of normalized singular-value energy, exponentiated) does match. `api.py` correctly scopes CORS `allow_origins=["*"]` to the local-dev standalone server with a `# local dev server only` docstring. Smoke-import: `runboard ok ['Run', 'init', 'RunStore', 'default_store', 'RankMonitor', 'effective_rank', 'r2d_curvature']`.

- 🟢 **`datalab` dependency hygiene is exemplary** — `datalab/pyproject.toml:12-22` puts heavy adapters behind `pdf, vector, chunking, observe` extras with a comment block explaining why (litellm→tokenizers, marker→transformers pins mustn't constrain the training env). Core deps are just `pydantic, httpx, numpy`. `store.py:70-102` lazy-imports `qdrant_client` inside `__init__` and `open_store` falls back to `LocalVectorStore` on `ImportError` (`store.py:111-114`). `pipeline.py` emits a point-in-time `DatasetManifest` + OKF card and skips the card on all-failed runs (`pipeline.py:140`). Smoke-import: `datalab ok ['chunk_text', 'CollectionRun', 'run_collection', 'Chunk', 'DatasetManifest', 'FinancialMetrics', 'SourceDoc']`.

- 🟢 **`omni-sim` has zero core deps and enforces sim-only** — `omni_sim/pyproject.toml:7` `dependencies = []`. `sim_env.py:57-58` raises `PermissionError("live capital execution is blocked in simulation mode (Spec 0013)")` and again at `sim_env.py:62-66` if the platform registry isn't `simulationOnly`/has `liveCapital` constraint. `_has_lookahead` (`sim_env.py:140-144`) rejects look-ahead-biased fixtures. `kag.py` appends 4-tuple `(s_t, a_t, o_t, v_t)` trajectories to `data/omni/trajectories.jsonl`. Smoke-import: `omni_sim ok ['get_platform', 'list_platforms', 'run_simulation', 'smart_search']`.

- 🟢 **`asn-engine` core deps are appropriate and `model` is opt-in** — `asn-engine/pyproject.toml:7-9` core is `torch, numpy` (the engine is a torch package; torch is not a "heavy adapter" here, it's the runtime). `transformers, tokenizers` are behind the `model` extra (`pyproject.toml:12`). `__init__.py` exports `effective_rank, newton_schulz, three_tier_surgery, info_nce, cosine_contrastive, procrustes`. Smoke-import: `asn_engine ok ['effective_rank', 'newton_schulz', 'three_tier_surgery', 'info_nce', 'cosine_contrastive', 'procrustes']`.

- 🟢 **No direct-Postgres bypass from any site** — repo-wide grep for `psycopg|asyncpg|DATABASE_URL|@neondatabase|pgvector|sqlalchemy` across `apps/` resolves to: `apps/sites/research-rag/components/ArxivExamDemo.tsx:7` (UI string "chunks in pgvector"), `apps/sites/dumbmodel/components/ComparePanel.tsx:115` (UI string "Live pgvector"), and `apps/synthorg/agent/connections/neon.ts` (a typed integration descriptor for the fleet agent, not a site; `via: "vercel-connect-oidc"`). No site imports a Postgres driver or `DATABASE_URL`. The tenancy rule holds at the site layer.

- 🟢 **Cross-package dependency graph is acyclic and coherent** — `cli → synth-core`; `ui-fleet → fleet`; `ui-fleet/routes → ui-fleet/site-api`; `eval-harness → asn-engine` (workspace); `core-api → omni-sim, runboard, datalab` (workspace). No back-edges, no `fleet → ui-fleet` or `synth-core → cli`. The one missing edge is `core-api → eval-harness` (see 🔴 above).

## Risk
- **🔴 eval-harness workspace gap (severity: high, blast radius: release pipeline)** — A clean `uv sync` does not install `eval_harness`. The first `/v1/eval/run` or `/v1/eval/{mv}/gates` call in a freshly-provisioned env (Railway container, new contributor box, CI matrix without the Dockerfile path hack) raises `ModuleNotFoundError` at `app/services/eval.py:47`. Deploy gates are spec 0008's release gate; this is the single most load-bearing package and the most fragile wiring in the repo.
- **🔴 core-api undeclared eval-harness dep (severity: high)** — Same root cause, second symptom. Even if eval-harness is installed by some side channel, the undeclared dependency means `uv pip tree` / lockfile / reproducibility are all lying. Renovate/Dependabot won't propose eval-harness bumps for core-api.
- **🟡 synth-core bypass (severity: medium, blast radius: observability + spec 0006 contract)** — Site→core-api traffic is untraced. The trace store (the "Operations Ledger" / `auto_research_ledger`) is missing every search, diagnose, feedback, models, ledger, hill-climb, and bd-queue call from every site. Hill-climb routing decisions (Spec 0012) are made on this telemetry; decisions are blind to the largest traffic source. The rule is documented in `CLAUDE.md` and `AGENTS.md` but not enforced by a lint or import boundary.
- **🟡 MRL gate stub (severity: medium)** — A model with broken Matryoshka truncation (rank collapse at `truncateDims=8`) ships as `allPassed: true`. Spec 0008 lists MRL as a gate; the whitepaper promotes edge-serving tiering as a feature. A silent regression here undermines the tier-comparison demos on `dumbmodel` and `research-rag`.
- **🟡 eval-harness has no tests (severity: medium)** — nDCG, retrieval scoring, and the gate booleans are pure functions and trivially testable; absence is a process smell, not a complexity barrier. If someone weakens `MIN_NDCG10` or flips gate logic, nothing catches it.
- **🟡 Dead code (severity: low)** — `FleetNavMobile.tsx` and `parse-args.ts` add maintenance surface and confuse future contributors who'll assume they're wired. The barrel-export gap for `FleetNavMobile` also quietly violates the `use-design-system` skill's "keep barrel current" rule.
- **🟡 `StatusPill` site-local (severity: low)** — Imminent duplication risk once a second site needs a status pill; minor today, compound cost as the fleet grows.

## Recommendation
1. **Enroll `eval-harness` in the uv workspace.** Add `packages/eval-harness` to `pyproject.toml:9-17` `members` and `eval-harness = { workspace = true }` to `[tool.uv.sources]`. Then add `eval-harness` to `services/core-api/pyproject.toml:6-13` `dependencies` (with `eval-harness = { workspace = true }` in its `[tool.uv.sources]`). Re-run `uv sync --all-packages` and confirm `uv run --no-sync python -c "import eval_harness"` succeeds. Delete the `sys.path.insert(...)` hacks in `scripts/tenant_baseline.py:24` and `scripts/engine_proof.py:27`.
2. **Wire sites through `synth-core` (or explicitly accept the bypass).** Either (a) refactor `packages/ui-fleet/src/site-api.ts` to build a `Synth` client via `synthFromEnv("site:<siteId>")` and route `siteSearch/siteModels/siteLedger/siteFeedback/siteBdQueue/siteHillClimb/siteDiagnose` through `Synth.vector.search/.model.list/.ledger.tail/.ledger.record/.evals.run/.model.embed`, so every call carries trace headers; or (b) write a one-paragraph ADR recording that site route handlers intentionally use raw `fetch` because they're server-side Next.js and the `Synth` client is for agent/CLI contexts only, and update `AGENTS.md` rule 1 wording to match. Pick (a) — it closes the trace gap and is a ~40-line change.
3. **Implement the MRL gate.** In `eval_harness/runner.py`, after `er = effective_rank(z)`, also compute effective rank on the truncated embedding matrix at the recipe's `truncateDims` (default 8) and have `gates.py:compute_gates` accept `mrlTruncatedRank` + `mrlTolerance` and return `"mrlWithinTolerance": mrlTruncatedRank >= mrlTolerance * er`. Set `mrlTolerance = 0.6` as the initial threshold (match `BASELINE_EFFECTIVE_RANK` style). Add a unit test in a new `packages/eval-harness/tests/test_gates.py`.
4. **Add `eval-harness/tests/`.** Cover `metrics.ndcg_at_k` (perfect ranking → 1.0, empty → 0.0, k truncation), `metrics.retrieval_scores` (cosine ordering), and `gates.compute_gates` (all-true, all-false, MRL-stub-then-real). Mirror the style of `packages/runboard/tests/test_runboard.py`.
5. **Either export `FleetNavMobile` from `ui-fleet/src/index.ts` and use it in `FleetShell.tsx` (which currently duplicates the toggle/nav pattern at `FleetShell.tsx:33-55`), or delete `FleetNavMobile.tsx`.** The duplication between `FleetShell` and `FleetNavMobile` is a smell — pick one.
6. **Either use `parseGlobalFlags` in `cli/src/index.ts` (call it on `process.argv.slice(2)` before dispatching, so `synth --org hub data ingest ...` works) or delete `parse-args.ts`.** The `--org` flag is the documented multi-tenant entry point; wiring it is a ~5-line change and unlocks `synth fleet context [siteId]`-style per-org commands.
7. **Promote `StatusPill` to `ui-fleet`** once a second site needs it; for now, leave a `// TODO: promote to ui-fleet when 2nd consumer appears` comment in `apps/sites/training-console/components/StatusPill.tsx` to flag the tripwire.
8. **No action needed** on fleet barrel, runboard, datalab, omni-sim, asn-engine, or the no-Postgres-bypass property — all healthy.
